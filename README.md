# 🚀 Система управления банковскими картами

Backend-приложение на Java Spring Boot для управления пользователями, банковскими картами и переводами между своими картами.

Проект реализует REST API с аутентификацией через JWT, ролевым доступом, защитой чувствительных данных, миграциями базы данных через Liquibase и документацией API через Swagger/OpenAPI.

## Стартовая структура и организация проекта

Проект реализован с разделением по основным слоям приложения. Код разложен по отдельным пакетам, чтобы было проще понимать, где находится контроллер, где бизнес-логика, где работа с базой данных, а где вспомогательные компоненты.

Основная структура проекта:

- `controller` — REST-контроллеры. Они принимают HTTP-запросы, проверяют входные данные и передают выполнение в сервисы;
- `service` — интерфейсы сервисов. В них описаны основные действия приложения;
- `service.impl` — реализации сервисов. Здесь находится основная бизнес-логика: работа с пользователями, картами, переводами и статусами;
- `service.finder` — отдельные компоненты для поиска сущностей. Они помогают не дублировать один и тот же код поиска пользователя, карты или перевода в разных сервисах;
- `service.scheduler` — фоновые задачи, например проверка карт с истёкшим сроком действия;
- `repository` — репозитории Spring Data JPA для работы с базой данных;
- `entity` — JPA-сущности, которые описывают таблицы базы данных;
- `dto` — объекты запросов и ответов API. Они отделяют внешний формат данных от внутренних сущностей приложения;
- `mapstruct` — мапперы для преобразования сущностей в DTO и обратно;
- `security` — настройки авторизации, JWT-фильтр и компоненты Spring Security;
- `exception` — пользовательские исключения и общий обработчик ошибок;
- `util` — вспомогательные классы, например для шифрования, маскирования и генерации данных карты;
- `resources/db/migration` — Liquibase-миграции для создания и изменения структуры базы данных;
- `docs` — OpenAPI-спецификация проекта;
- `postman` — Postman Collection для ручной проверки API;
- `src/test` — тесты приложения. В проекте есть unit-тесты сервисов, тесты генерации данных карты, тесты JWT-сервиса и тесты контроллеров через MockMvc.

Также в проект добавлены файлы для запуска и проверки приложения:

- `Dockerfile` — описывает сборку Spring Boot приложения в Docker-образ. В проекте используется multi-stage build: на первом этапе приложение собирается, а на втором этапе запускается уже готовый `.jar` файл;
- `docker-compose.yml` — поднимает PostgreSQL и Spring Boot приложение вместе. Для базы данных настроен volume, чтобы данные сохранялись между перезапусками контейнера. Также добавлен healthcheck для PostgreSQL, чтобы приложение запускалось после готовности базы;
- `Makefile` — содержит короткие команды для удобного запуска проекта, тестов и просмотра логов. Например, вместо длинных Docker-команд можно использовать `make dev`, `make test`, `make down`, `make logs-app`.

Сервисный слой сделан через интерфейсы и отдельные реализации. Например, контроллер работает с интерфейсом `AdminCardService`, а конкретная логика находится в `AdminCardServiceImpl`. Это делает код более гибким: контроллер не привязан напрямую к конкретному классу, а реализацию сервиса при необходимости можно заменить или расширить.

Отдельное внимание уделено тестированию. Основная бизнес-логика покрыта unit-тестами сервисов: создание и обновление пользователей, управление ролями и статусами, создание карт, блокировка и активация карт, переводы между картами и обработка ошибок. Для REST-слоя добавлены MockMvc-тесты контроллеров, которые проверяют HTTP-статусы, валидацию входных данных, JSON-ответы и вызовы сервисного слоя. Также отдельно проверяется работа JWT-сервиса и генерация данных карты.

Такое разделение помогает держать проект понятным. Контроллеры не содержат бизнес-логику, сервисы отвечают за правила работы приложения, репозитории работают с базой данных, а DTO используются для общения с клиентом через API.


## 📝 Описание задачи

Проект представляет собой backend-приложение на **Java** и **Spring Boot** для управления банковскими картами.

В системе реализованы основные операции, необходимые для работы с картами:

- создание банковских карт;
- управление статусами карт;
- просмотр карт;
- просмотр баланса карты;
- запрос блокировки карты пользователем;
- обработка блокировки карты администратором;
- переводы между собственными картами пользователя;
- просмотр истории переводов.

Приложение разделяет возможности администратора и обычного пользователя.

**Администратор** может создавать карты для пользователей, просматривать список карт, управлять статусами карт, обрабатывать заявки на блокировку и логически удалять карты.

**Пользователь** может просматривать только свои карты, смотреть баланс, отправлять запрос на блокировку карты и выполнять переводы только между своими картами.

При создании карты номер карты и срок действия генерируются сервером автоматически. Полный номер карты не возвращается через API: он хранится в зашифрованном виде, а наружу отдаётся только маска вида `**** **** **** 1234`.

Для проверки уникальности номера карты используется hash, который хранится в базе данных и не возвращается клиенту.

Таким образом, приложение покрывает основные требования задачи:

- создание и управление картами;
- просмотр карт;
- переводы между собственными картами пользователя.
## 💳 Атрибуты карты

В проекте реализована модель банковской карты с основными атрибутами, указанными в техническом задании.

Карта содержит номер карты, владельца, срок действия, статус и баланс.

Полный номер карты не хранится и не возвращается через API в открытом виде. При создании карты номер генерируется сервером автоматически, после чего сохраняется в базе данных в зашифрованном виде. Для отображения используется только маска номера карты: `**** **** **** 1234`.

Также в базе отдельно хранится hash номера карты. Он используется только для внутренней проверки уникальности сгенерированного номера и не возвращается клиенту через API.

Владелец карты указывается при создании карты и сохраняется в поле `ownerName`.

Срок действия карты рассчитывается сервером автоматически при создании карты. Клиент не передаёт срок действия вручную, что снижает риск некорректных данных.

Для карты поддерживаются следующие статусы:

- `ACTIVE` — карта активна;
- `BLOCKED` — карта заблокирована;
- `EXPIRED` — срок действия карты истёк.

Баланс карты хранится в `BigDecimal`, что подходит для денежных значений и позволяет избежать ошибок округления, характерных для `double` или `float`.

Таким образом, все атрибуты карты из технического задания реализованы. Дополнительно в проекте добавлены шифрование полного номера карты, hash для проверки уникальности и безопасное отображение только маскированного номера.
