# Bank Card Management API

Backend-приложение на **Java** и **Spring Boot** для управления банковскими картами, пользователями и переводами между собственными картами пользователя.

Проект реализует REST API с авторизацией через JWT, разграничением доступа по ролям `ADMIN` и `USER`, шифрованием номера карты, маскированием отображаемого номера, переводами между картами, фильтрацией, постраничной выдачей, Swagger/OpenAPI-документацией, Docker-запуском, автоматическими тестами и Postman Collection для ручной проверки API.

## 📝 Описание проекта

Проект разработан как backend-система для управления банковскими картами.

В приложении реализованы основные возможности:

- создание и управление банковскими картами;
- просмотр карт администратором и пользователем;
- просмотр баланса карты;
- переводы между собственными картами пользователя;
- запрос пользователем блокировки карты;
- обработка блокировки карты администратором;
- управление пользователями;
- логическое удаление пользователей и карт;
- фильтрация и постраничная выдача списков;
- OpenAPI-документация;
- автоматические тесты сервисного и controller-слоя;
- Postman Collection для ручной проверки API;
- запуск приложения через Docker Compose.

Приложение разделяет административные и пользовательские сценарии.

**Администратор** управляет пользователями и картами: создаёт пользователей, создаёт карты, блокирует и активирует карты, просматривает списки пользователей и карт, видит заявки на блокировку через фильтр `blockRequested`, а также может работать с логически удалёнными записями.

**Пользователь** работает только со своими данными: просматривает свои карты, смотрит баланс, создаёт запрос на блокировку карты и выполняет переводы только между своими картами.

Номер карты и срок действия генерируются сервером автоматически. Клиент не передаёт эти значения вручную. Полный номер карты не возвращается через API: он хранится в зашифрованном виде, а наружу отдаётся только маска вида `**** **** **** 1234`.

Для проверки уникальности номера карты используется hash полного номера. Hash хранится только в базе данных и не возвращается клиенту через API.

Для проверки проекта добавлена Postman Collection, расположенная в папке `postman`. Она содержит основные позитивные и негативные сценарии: авторизацию, управление пользователями, создание карт, просмотр карт, запрос блокировки, переводы между картами и проверки ошибок.

Автоматические тесты покрывают ключевую бизнес-логику сервисов, работу JWT, генерацию данных карты, обработку ошибок и REST-контроллеры через MockMvc.

## 💳 Атрибуты карты

В проекте реализована модель банковской карты со всеми основными атрибутами из технического задания.

Карта содержит:

- номер карты;
- владельца карты;
- срок действия;
- статус;
- баланс.

Полный номер карты не возвращается через API в открытом виде. При создании карты номер генерируется сервером автоматически, после чего сохраняется в базе данных в зашифрованном виде. Для отображения используется только маска номера карты: `**** **** **** 1234`.

Дополнительно для номера карты хранится hash. Он используется для внутренней проверки уникальности сгенерированного номера карты и не возвращается клиенту через API.

Владелец карты хранится в поле `ownerName`. Это имя владельца, зафиксированное для конкретной карты.

Срок действия карты рассчитывается сервером автоматически при создании карты. Клиент не передаёт срок действия вручную, что снижает риск создания карты с некорректной датой.

Для карты поддерживаются следующие статусы:

- `ACTIVE` — карта активна;
- `BLOCKED` — карта заблокирована;
- `EXPIRED` — срок действия карты истёк.

Баланс карты хранится в `BigDecimal`, что подходит для работы с денежными значениями и позволяет избежать ошибок округления, характерных для `double` и `float`.

## ✅ Аутентификация и авторизация

В проекте реализована аутентификация и авторизация через **Spring Security** и **JWT**.

Пользователь проходит авторизацию через endpoint:

`POST /api/auth/login`

После успешной авторизации сервер возвращает JWT-токен. Далее клиент передаёт этот токен в заголовке `Authorization`:

`Authorization: Bearer <token>`

Для доступа к API используются две роли:

- `ADMIN` — администратор;
- `USER` — обычный пользователь.

Доступ к административным endpoint'ам ограничен ролью `ADMIN`. К таким endpoint'ам относятся управление пользователями, создание и управление картами, просмотр административных списков пользователей и карт.

Пользовательские endpoint'ы доступны авторизованным пользователям с ролью `USER` или `ADMIN`. Пользователь может работать только со своими картами и своими переводами.

JWT используется для stateless-аутентификации: сервер не хранит HTTP-сессию, а определяет пользователя по токену, который приходит в каждом защищённом запросе.

В проекте также реализован сервис текущего пользователя, который получает данные авторизованного пользователя из security context. Это используется в пользовательских сценариях, например при просмотре своих карт или переводов.

## ✅ Возможности

В проекте реализовано разделение функциональности по ролям: `ADMIN` и `USER`.

### Администратор

Администратор имеет доступ к административной части API и может управлять пользователями и банковскими картами.

Возможности администратора:

- создавать новых пользователей;
- получать список пользователей с фильтрацией и постраничной выдачей;
- получать данные конкретного пользователя по идентификатору;
- обновлять данные пользователя;
- изменять роль пользователя;
- блокировать пользователя;
- активировать пользователя;
- логически удалять пользователя;
- создавать банковские карты для пользователей;
- получать список всех карт с фильтрацией и постраничной выдачей;
- получать данные конкретной карты по идентификатору;
- блокировать карту;
- активировать карту;
- логически удалять карту;
- просматривать карты, по которым пользователь запросил блокировку, через фильтр `blockRequested=true`.

Для административного списка пользователей реализованы фильтры:

- `email`;
- `phone`;
- `role`;
- `status`;
- `includeDeleted`.

Для административного списка карт реализованы фильтры:

- `status`;
- `includeDeleted`;

- `ownerName`;
- `expirationDateFrom`;
- `expirationDateTo`;

- отправлять запрос на блокировку своей карты;
- выполнять переводы между своими активными картами;
- просматривать историю своих переводов с фильтрацией и постраничной выдачей;
- получать данные конкретного своего перевода.

Для списка своих карт реализованы фильтры:

Для списка своих переводов реализованы фильтры:

- `status`;
- `fromCardLast4`;
- `toCardLast4`;
- `amountFrom`;
- `amountTo`;

## ✅ API

В проекте реализован REST API для управления пользователями, банковскими картами и переводами между собственными картами пользователя.

### Карты

Для карт реализованы основные CRUD-операции и дополнительные действия:

- создание карты администратором;
- получение списка карт администратором;
- получение конкретной карты по идентификатору;
- блокировка карты администратором;
- активация карты администратором;
- логическое удаление карты;
- просмотр своих карт пользователем;
- просмотр баланса своей карты;
- запрос блокировки карты пользователем.

Создание карты выполняется через административный endpoint. При создании карты сервер самостоятельно генерирует номер карты и срок действия. Клиент передаёт только пользователя и начальный баланс.

### Переводы

Пользователь может выполнять переводы только между своими картами.

При создании перевода проверяется:

- что карта списания принадлежит текущему пользователю;
- что карта зачисления принадлежит текущему пользователю;
- что карты не совпадают;
- что обе карты активны;
- что сумма перевода корректна;
- что на карте списания достаточно средств.

После успешного перевода баланс карты списания уменьшается, а баланс карты зачисления увеличивается.

### Фильтрация и постраничная выдача

Для списковых endpoint'ов реализованы фильтрация и постраничная выдача.

Пагинация работает через параметры:

- `page` — номер страницы;
- `size` — размер страницы.

Ответ возвращается в едином формате `PageResponseDTO`:

- `items` — список элементов текущей страницы;
- `page` — номер текущей страницы;
- `size` — размер страницы;
- `total_elements` — общее количество найденных элементов;
- `total_pages` — общее количество страниц;
- `first` — является ли страница первой;
- `last` — является ли страница последней.

Фильтрация реализована через Spring Data JPA `Specification`. Это позволяет динамически собирать запрос к базе данных только по тем параметрам, которые были переданы клиентом.

### Валидация и ошибки

Входные данные проверяются через Bean Validation:

- `@NotNull`;
- `@NotBlank`;
- `@Positive`;
- `@DecimalMin`;
- другие ограничения DTO.

В проекте реализована централизованная обработка ошибок через global exception handler.

API возвращает понятные HTTP-статусы:

- `400 Bad Request` — некорректные входные данные;
- `401 Unauthorized` — пользователь не авторизован;
- `403 Forbidden` — недостаточно прав;
- `404 Not Found` — сущность не найдена;
- `409 Conflict` — бизнес-конфликт, например попытка создать пользователя с уже занятым email или выполнить некорректную операцию с картой.

## ✅ Безопасность

В проекте реализованы механизмы защиты данных и разграничения доступа.

### Шифрование данных

Полный номер банковской карты не хранится в базе данных в открытом виде. При создании карты сервер генерирует номер карты, после чего сохраняет его в зашифрованном виде.

Для проверки уникальности номера карты дополнительно используется hash полного номера. Hash хранится только в базе данных, используется внутри backend-приложения и не возвращается клиенту через API.

### Ролевой доступ

Доступ к API разделён по ролям:

- `ADMIN` — администратор;
- `USER` — обычный пользователь.

Административные endpoint'ы доступны только пользователям с ролью `ADMIN`. Пользовательские endpoint'ы доступны авторизованным пользователям и ограничены текущим пользователем: пользователь может просматривать только свои карты и выполнять переводы только между своими картами.

Авторизация реализована через Spring Security и JWT. После успешного входа клиент получает токен и передаёт его в заголовке `Authorization`.

### Маскирование номеров карт

Полный номер карты не возвращается в API-ответах. Для отображения используется маска:

`**** **** **** 1234`

Для этого в базе отдельно хранится только последняя часть номера карты — последние 4 цифры. Они используются для формирования безопасного отображения номера карты.

## 🗄️ База данных и миграции

В проекте используется **PostgreSQL** в качестве основной базы данных.

Схема базы данных управляется через **Liquibase**. Все изменения структуры БД описаны в миграциях и хранятся в директории:

`src/main/resources/db/migration`

При запуске приложения Liquibase автоматически применяет миграции и создаёт необходимые таблицы, ограничения и связи.

В базе данных хранятся основные сущности проекта:

- пользователи;
- банковские карты;
- переводы между картами.

Для важных полей настроены ограничения на уровне базы данных, например уникальность email, телефона пользователя и hash номера карты.

Hash номера карты используется для проверки уникальности сгенерированного номера. Полный номер карты хранится в зашифрованном виде, а наружу через API не возвращается.

Для запуска базы данных в Docker Compose настроен сервис PostgreSQL. Данные PostgreSQL сохраняются в Docker volume, поэтому база не теряется при обычном перезапуске контейнеров.

## ✅ Развёртывание и тестирование

В проекте подготовлена dev-среда для локального запуска и проверки приложения.

### Dockerfile

Для сборки приложения добавлен `Dockerfile`.

Он описывает создание Docker-образа для Spring Boot приложения. В проекте используется multi-stage build:

- на первом этапе приложение собирается через Maven;
- на втором этапе запускается уже готовый `.jar` файл.

Такой подход позволяет не хранить Maven и исходный код внутри финального runtime-образа, а запускать только собранное Java-приложение.

### Docker Compose

Для запуска приложения используется `docker-compose.yml`.

Docker Compose поднимает два основных сервиса:

- `postgres` — база данных PostgreSQL;
- `app` — Spring Boot приложение.

Для PostgreSQL настроен volume, чтобы данные базы сохранялись между обычными перезапусками контейнеров. Также добавлен healthcheck, чтобы приложение запускалось только после готовности базы данных.

Запуск проекта:

`docker compose up --build`

Или через Makefile:

`make dev`

- шифрование и маскирование номера карты;
- блокировка, активация и логическое удаление карт;
- запрос пользователем блокировки карты;
- переводы между своими картами;

- JWT-сервис;
