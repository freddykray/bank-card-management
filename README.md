## 🚀 Соответствие техническому заданию

Проект представляет собой backend-приложение на **Java 17** и **Spring Boot** для управления банковскими картами, пользователями и переводами между собственными картами пользователя.

Приложение реализует основные требования технического задания: создание и управление картами, просмотр карт, переводы между своими картами, ролевой доступ, JWT-аутентификацию, шифрование номера карты, маскирование номера, фильтрацию, постраничную выдачу, работу с PostgreSQL, миграции Liquibase, Swagger/OpenAPI-документацию, Docker-запуск и автоматические тесты.

## 📁 Стартовая структура

Проект реализован с разделением по слоям и зонам ответственности. Код размещён в соответствующих пакетах:

- `controller` — REST-контроллеры;
- `service` — интерфейсы сервисного слоя;
- `service.impl` — реализации сервисов с основной бизнес-логикой;
- `service.finder` — компоненты для централизованного поиска сущностей;
- `repository` — Spring Data JPA репозитории;
- `entity` — JPA-сущности;
- `dto` — объекты запросов и ответов API;
- `mapstruct` — мапперы между entity и DTO;
- `security` — JWT, Spring Security и настройка доступа;
- `exception` — пользовательские исключения и глобальная обработка ошибок;
- `specification` — динамическая фильтрация через Spring Data JPA Specification;
- `util` — вспомогательные компоненты;
- `resources/db/migration` — Liquibase-миграции;
- `docs` — OpenAPI-спецификация;
- `postman` — Postman Collection для ручной проверки API.

Сервисный слой построен через интерфейсы и реализации. Контроллеры зависят от интерфейсов сервисов, а не от конкретных классов реализации. Это делает код более гибким, тестируемым и удобным для расширения.

## 📝 Описание задачи

В проекте реализована backend-система для управления банковскими картами.

Основные возможности приложения:

- создание и управление банковскими картами;
- просмотр карт администратором и пользователем;
- просмотр баланса карты;
- переводы между собственными картами пользователя;
- запрос блокировки карты пользователем;
- обработка блокировки карты администратором;
- управление пользователями;
- логическое удаление пользователей и карт;
- фильтрация и постраничная выдача списков.

Администратор управляет пользователями и картами. Пользователь работает только со своими картами и своими переводами.

## 💳 Атрибуты карты

В проекте реализована модель банковской карты со всеми основными атрибутами из технического задания.

Карта содержит:

- номер карты;
- владельца карты;
- срок действия;
- статус;
- баланс.

Полный номер карты не возвращается через API в открытом виде. При создании карты номер генерируется сервером автоматически, после чего сохраняется в базе данных в зашифрованном виде.

Для отображения используется только маска номера карты: `**** **** **** 1234`.

Дополнительно для номера карты хранится hash. Он используется для внутренней проверки уникальности сгенерированного номера карты и не возвращается клиенту через API.

Срок действия карты рассчитывается сервером автоматически при создании карты. Клиент не передаёт срок действия вручную.

Для карты поддерживаются статусы:

- `ACTIVE` — карта активна;
- `BLOCKED` — карта заблокирована;
- `EXPIRED` — срок действия карты истёк.

Баланс карты хранится в `BigDecimal`, что подходит для работы с денежными значениями.

## ✅ Аутентификация и авторизация

В проекте реализована аутентификация и авторизация через **Spring Security** и **JWT**.

Пользователь авторизуется через endpoint:

`POST /api/auth/login`

После успешной авторизации сервер возвращает JWT-токен. Далее клиент передаёт токен в заголовке:

`Authorization: Bearer <token>`

В системе используются две роли:

- `ADMIN`;
- `USER`.

Административные endpoint'ы доступны только пользователям с ролью `ADMIN`.

Пользовательские endpoint'ы доступны авторизованным пользователям и ограничены текущим пользователем. Пользователь может просматривать только свои карты, смотреть баланс только своих карт и выполнять переводы только между своими картами.

Для определения текущего пользователя используется security context.

## ✅ Возможности

### Администратор

Администратор может:

- создавать пользователей;
- получать список пользователей с фильтрацией и постраничной выдачей;
- получать данные конкретного пользователя;
- обновлять данные пользователя;
- изменять роль пользователя;
- блокировать пользователя;
- активировать пользователя;
- логически удалять пользователя;
- создавать банковские карты для пользователей;
- получать список всех карт с фильтрацией и постраничной выдачей;
- получать данные конкретной карты;
- блокировать карту;
- активировать карту;
- логически удалять карту;
- просматривать заявки пользователей на блокировку карт через фильтр `blockRequested=true`.

### Пользователь

Пользователь может:

- просматривать список своих карт с фильтрацией и постраничной выдачей;
- получать данные конкретной своей карты;
- смотреть баланс своей карты;
- отправлять запрос на блокировку своей карты;
- выполнять переводы между своими активными картами;
- просматривать историю своих переводов с фильтрацией и постраничной выдачей;
- получать данные конкретного своего перевода.

## ✅ API

В проекте реализован REST API для работы с пользователями, картами и переводами.

### Карты

Для карт реализованы:

- создание карты;
- получение списка карт;
- получение карты по идентификатору;
- блокировка карты;
- активация карты;
- логическое удаление карты;
- просмотр своих карт пользователем;
- просмотр баланса своей карты;
- запрос блокировки карты пользователем.

При создании карты сервер самостоятельно генерирует номер карты и срок действия. Клиент передаёт только данные, необходимые для создания карты.

### Переводы

Пользователь может выполнять переводы только между своими картами.

При переводе проверяется:

- карта списания принадлежит текущему пользователю;
- карта зачисления принадлежит текущему пользователю;
- карты не совпадают;
- обе карты активны;
- сумма перевода корректна;
- на карте списания достаточно средств.

После успешного перевода баланс карты списания уменьшается, а баланс карты зачисления увеличивается.

### Фильтрация и постраничная выдача

Для списковых endpoint'ов реализованы фильтрация и постраничная выдача.

Пагинация работает через параметры:

- `page` — номер страницы;
- `size` — размер страницы.

Ответ возвращается в едином формате `PageResponseDTO`:

- `items` — список элементов текущей страницы;
- `page` — номер текущей страницы;
- `size` — размер страницы;
- `total_elements` — общее количество найденных элементов;
- `total_pages` — общее количество страниц;
- `first` — является ли страница первой;
- `last` — является ли страница последней.

Фильтрация реализована через Spring Data JPA `Specification`. Запрос к базе данных собирается динамически только по тем параметрам, которые передал клиент.

Для административного списка пользователей поддерживаются фильтры:

- `email`;
- `phone`;
- `role`;
- `status`;
- `includeDeleted`.

Для административного списка карт поддерживаются фильтры:

- `status`;
- `includeDeleted`;
- `last4`;
- `ownerName`;
- `expirationDateFrom`;
- `expirationDateTo`;
- `balanceFrom`;
- `balanceTo`;
- `blockRequested`.

Для списка своих карт поддерживаются фильтры:

- `status`;
- `last4`;
- `expirationDateFrom`;
- `expirationDateTo`;
- `balanceFrom`;
- `balanceTo`;
- `blockRequested`.

Для списка своих переводов поддерживаются фильтры:

- `status`;
- `fromCardLast4`;
- `toCardLast4`;
- `amountFrom`;
- `amountTo`;
- `createdFrom`;
- `createdTo`.

### Валидация и ошибки

Входные данные проверяются через Bean Validation.

В проекте используются:

- `@NotNull`;
- `@NotBlank`;
- `@Positive`;
- `@DecimalMin`;
- другие ограничения DTO.

Ошибки обрабатываются централизованно через global exception handler.

API возвращает понятные HTTP-статусы:

- `400 Bad Request` — некорректные входные данные;
- `401 Unauthorized` — пользователь не авторизован;
- `403 Forbidden` — недостаточно прав;
- `404 Not Found` — сущность не найдена;
- `409 Conflict` — бизнес-конфликт.

## ✅ Безопасность

В проекте реализованы основные механизмы защиты данных.

### Шифрование данных

Полный номер карты не хранится в базе данных в открытом виде. Перед сохранением номер карты шифруется.

### Ролевой доступ

Доступ к API ограничен ролями `ADMIN` и `USER`.

Администратор имеет доступ к административным операциям. Пользователь имеет доступ только к своим картам и переводам.

### Маскирование номеров карт

Полный номер карты не возвращается в API-ответах.

Для отображения используется маска:

`**** **** **** 1234`

Для формирования маски в базе отдельно хранятся последние 4 цифры номера карты.

## ✅ Работа с БД

В проекте используется **PostgreSQL**.

Схема базы данных управляется через **Liquibase**. Все миграции находятся в директории:

`src/main/resources/db/migration`

При запуске приложения Liquibase автоматически применяет миграции и создаёт необходимые таблицы, ограничения и связи.

В базе данных хранятся:

- пользователи;
- банковские карты;
- переводы между картами.

На уровне базы данных настроены ограничения уникальности для важных полей, например email, телефона пользователя и hash номера карты.
